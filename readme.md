# TRPG-Agent 项目介绍

## 项目背景

### 跑团是什么

TRPG，也就是桌上角色扮演游戏，常被叫作“跑团”。玩家扮演冒险者，主持人 DM 负责描述世界、扮演 NPC、解释规则，并根据玩家的行动推动剧情。一次典型的跑团流程大概是：玩家说出想做什么，DM 判断这件事是否需要检定，玩家掷骰，最后 DM 根据结果描述后果。

### 纯文本 AI 跑团的问题

大语言模型让“AI 当 DM”变得可行，但纯文本 AI 跑团有几个明显问题。第一，它容易过度顺从玩家，比如玩家说“我一拳打死巨龙”，模型可能真的顺着写下去，游戏就失去了规则和挑战。第二，它不擅长长期稳定地记住生命值、法术位、状态效果、敌人位置等数据。第三，纯聊天窗口缺少跑团里很重要的仪式感，比如掷骰、看角色卡、观察战场位置。

### 这个项目的目标

TRPG-Agent 想解决的就是这些问题：让 AI 负责叙事和理解玩家意图，把硬规则、数值变化和状态保存交给后端程序处理。这样既保留自然语言跑团的自由度，也让游戏过程有更清楚的规则边界。

## 这个项目是什么

### 一句话概括

TRPG-Agent 是一个基于 Agent 架构的 Web 端 AI 跑团系统。目前它以 D&D 5e 风格的规则为基础，提供一个可以在浏览器里玩的 AI 跑团体验。

### 玩家如何参与

玩家可以直接用自然语言和 AI DM 对话，例如：

- “我选择法师角色，进入地牢。”
- “我观察房间里有没有陷阱。”
- “我向最近的敌人施放火焰箭。”
- “我结束回合。”

### AI 和系统如何分工

AI 会负责理解这些话背后的游戏意图。如果只是普通交流，它会直接描述剧情；如果涉及掷骰、角色状态、战斗、法术、移动距离等客观规则，它会调用后端工具来完成计算，再把结果转成玩家能读懂的叙事文本。

## 基本玩法

### 当前游戏闭环

目前的核心玩法已经围绕“探索 - 遭遇 - 战斗 - 结算”形成了一个基础闭环。

1. 玩家进入聊天界面，与 AI DM 开始对话。
2. 玩家可以加载预设角色卡，目前项目内已经准备了战士、法师、游荡者、牧师、术士、游侠、武僧、圣武士、邪术师、德鲁伊、吟游诗人、野蛮人等职业模板。
3. 在探索阶段，玩家通过自然语言行动。AI DM 会描述场景，也可以根据需要创建平面地图、放置单位、查询规则或要求检定。
4. 当进入战斗时，系统会生成敌人、排定先攻顺序，并切换到战斗流程。
5. 战斗中，攻击、伤害、护甲等级、生命值、法术位、状态效果、移动距离等内容由后端工具计算，不靠模型随口编。
6. 前端会显示角色面板、生命值变化、战术地图和 3D 骰子动画，让玩家能看到游戏状态的变化。
7. 如果玩家角色倒下，系统会暂停流程，让玩家选择复活继续或结束战斗。

### 当前体验定位

这个项目目前更像一个“有规则骨架的 AI 单人跑团原型”。它已经能支持基础游玩，但距离完整商业化跑团平台还有很多内容需要继续补齐。

## 目前已经实现的主要模块

### 1. Web 前端交互

前端使用 Vue 3、TypeScript 和 Vite 构建。主要界面是一个聊天式跑团页面，左侧是对话流，右侧是角色和战场信息面板。

已经实现的前端能力包括：

- 聊天界面：支持玩家输入、AI 回复、战斗消息、工具消息和流式更新。
- 角色面板：展示角色名称、职业、等级、HP、AC、六维属性、状态效果、资源、法术和武器。
- 生命值概览：战斗时可以切换到 HP 条视图，方便观察参战单位血量。
- 战术地图：展示当前平面地图、网格、玩家、敌人和其他单位的位置。
- 3D 骰子动画：使用 Three.js 和 cannon-es 做了一个 D20 掷骰表现，后端给出骰点后，前端用动画展示结果。
- 挂起动作弹窗：用于处理掷骰确认、玩家倒下选择、反应法术选择等需要玩家确认的节点。
- 会话历史恢复：重新进入已有会话时，可以恢复最近的聊天记录和角色、战斗、地图状态。

### 2. 后端 API 与流式通信

后端使用 FastAPI 提供接口，核心接口包括普通聊天、SSE 流式聊天和历史记录查询。

目前前端主要通过 SSE 流式接口接收后端事件。后端会把不同类型的事件拆开推送给前端，例如：

- AI 文本回复
- 工具执行结果
- 战斗行动结果
- 掷骰事件
- 角色、战斗、地图状态更新
- 等待玩家处理的挂起动作

这样前端不需要等整轮处理全部结束，玩家可以更快看到过程反馈。

### 3. LangGraph Agent 流程

项目后端用 LangGraph 组织 Agent 工作流。整体图结构包含路由节点、探索代理、战斗代理、工具节点、战斗收束节点和反应结算节点。

这里比较重要的一点是：工具执行没有自己重新造轮子，而是直接使用 LangGraph 官方提供的 `ToolNode`。模型负责选择工具，`ToolNode` 负责执行工具，状态更新通过 LangGraph 的 `Command` 回写。这让代码更贴近框架标准，也减少了手动拼接工具调用流程的复杂度。

Agent 目前分成两种模式：

- 探索模式：更关注剧情推进、环境描述、玩家自由行动和规则查询。
- 战斗模式：更关注回合推进、攻击、移动、施法、资源消耗和战斗结算。

系统会根据当前 `phase` 和战斗状态自动选择合适的代理节点。

### 4. 结构化游戏状态

项目没有把所有内容都塞进一段聊天文本里，而是定义了结构化的游戏状态。后端用 TypedDict 和 Pydantic 模型维护这些数据。

当前状态中比较核心的部分包括：

- 玩家角色：等级、HP、AC、能力值、资源、武器、已知法术、状态效果等。
- 战斗状态：回合数、参战单位、先攻顺序、当前行动者。
- 场景单位池：已生成但不一定正在战斗的单位。
- 平面空间：地图、网格大小、单位坐标、朝向和占位半径。
- 死亡单位档案：战斗结束后保留倒下单位，方便后续搜刮或剧情处理。
- 挂起反应：例如敌人攻击命中玩家后，等待玩家决定是否施放 `shield`。

这些状态会通过 LangGraph checkpointer 按会话持久化，避免每次对话都从零开始。

### 5. 角色与成长系统

项目内置了多种预设职业角色卡，可以快速开始游戏。角色状态可以被工具修改，例如调整 HP、AC、速度、能力值、资源、状态效果等。

目前成长系统已经有基础雏形，尤其是法师升级路径做得更完整一些，包括经验值、升级、法术位变化、新法术、奥术传承选择等。其他职业的模板已经存在，但完整升级表还没有完全展开。

### 6. 战斗系统

战斗系统是目前完成度较高的部分之一。它已经支持：

- 从 Open5e 怪物数据生成敌方单位。
- 自动为玩家和敌人投先攻。
- 回合制行动顺序。
- 攻击命中判定、暴击判定、伤害结算和 HP 扣减。
- 动作、附赠动作、反应、移动力等基础动作经济。
- 玩家和怪物共用统一的参战单位结构。
- 战斗结束后归档战斗摘要，避免后续上下文被大量战报撑满。
- 玩家倒下后的中断处理。

这部分的目标不是让模型“假装懂规则”，而是让模型在合适的时候调用工具，由后端完成实际判定。

### 7. 法术与状态效果

项目已经实现了一批常用法术和状态效果。它们不是只在文本里“说说而已”，而是会真实影响角色资源、战斗结果和状态面板。

目前支持的法术类型包括：

- 直接伤害类：例如火焰箭、火球术、燃烧之手、寒冰匕首等。
- 自动命中或特殊结算类：例如魔法飞弹。
- 防御和反应类：例如法师护甲、护盾术、镜影术。
- 控制类：例如人类定身术，以及造成减速、标记等效果的法术。
- 治疗类：例如疗伤术。

状态效果也被做成了独立模块。比如目盲、隐形、麻痹、魅惑、护盾、镜影、奥术结界等，都会进入角色或单位的结构化状态中。它们会影响护甲等级、移动速度、攻击优劣势、是否能行动、是否能反应等规则结果。

简单说，法术和状态不只是剧情描述，而是已经接入了后端的规则结算。

### 8. 平面地图与空间系统

项目已经有了基础的 2D 平面空间系统。后端负责保存地图和单位坐标，前端负责显示小地图。

目前支持：

- 创建和切换地图。
- 放置单位。
- 移动单位并扣除移动力。
- 测量两个单位之间的距离。
- 查询某个半径内的单位。
- 让怪物自动向目标靠近到合适距离。
- 范围法术根据地图坐标自动展开目标。

这让战斗不再只是“文字上说你在旁边”，而是有了可计算的位置关系。

### 9. 规则查询与 RAG

项目已经实现了一个混合检索模块，用于查询 D&D 相关规则材料。它结合了 BM25、向量检索和可选 rerank。这样当 AI DM 需要查规则时，可以通过工具检索规则片段，而不是完全依赖模型记忆。

目前这部分主要用于规则问答和裁决参考。后续如果要引入完整冒险模组，也可以继续沿用类似的检索和上下文注入方式。

### 10. 记忆与上下文管理

项目不是简单把所有历史消息都塞给模型。后端实现了上下文装配器，会把当前玩家状态、战斗状态、场景单位、地图、死亡单位等整理成 HUD 风格的状态快照，再插入模型上下文。

同时，项目还有后台记忆摄取流程，会把近期回合整理成情节摘要。战斗结束后也会把整段战斗折叠成摘要，减少后续对话的上下文压力。

## 技术架构简述

### 基础技术栈

项目整体是前后端分离结构。

- 前端：Vue 3 + TypeScript + Vite
- 3D 骰子：Three.js + cannon-es
- 后端：FastAPI + LangGraph + LangChain
- 模型调用：通过 LangChain 的 `ChatOpenAI` 接入兼容 OpenAI 接口的模型服务
- 工具调用：LangChain tool + LangGraph `ToolNode`
- 持久化：LangGraph SQLite checkpointer
- 规则检索：Chroma 向量库 + BM25 + 可选 rerank
- 怪物数据：Open5e API
- 骰子计算：`d20` Python 库
- 空间计算：Shapely

### 分工方式

整个架构的基本分工是：

- LLM 负责理解玩家意图、组织叙事、决定是否调用工具。
- 工具层负责规则、数值、状态、战斗和空间计算。
- LangGraph 负责把一轮对话拆成可控的状态流转。
- 前端负责把文本、状态、地图和骰子表现出来。

## 当前完成度与不足

### 已经跑通的部分

目前项目已经跑通了 AI 单人跑团的主要技术闭环：自然语言输入、Agent 决策、工具调用、角色状态、战斗结算、地图展示、流式反馈和会话记忆。

### 仍在补齐的部分

但它仍然是一个持续迭代中的项目。当前比较明显的不足包括：

- 主线剧情和可玩内容还不够完整，更多依赖 AI 即兴生成。
- 物品、背包和装备系统还没有真正成型。
- 多职业成长规则还不完整，目前法师相关成长更明确。
- 地图系统已经能支持坐标和距离，但障碍物、视线、地形等复杂规则还没有完全实现。
- NPC 还主要作为场景单位或敌我单位存在，没有形成长期记忆、关系网和独立目标。
- 多人联机还没有实现，目前更偏单人游玩。

## 后续设计方向

### 引入正式冒险模组

后续我希望把这个项目从“能跑起来的 AI 跑团原型”继续推进到“真的有主线、有队友、有长期可玩性”的系统。

首先，会考虑引入 D&D 冒险模组。现在的系统已经有规则检索、地图、战斗和记忆基础，下一步可以把正式冒险模组拆成地点、事件、NPC、遭遇、线索和奖励，让 AI DM 不只是即兴发挥，而是围绕一个稳定主线推进。这样能明显提高剧情连贯性和可玩性。

### 友方 NPC 与队友系统

其次，会引入更完整的友方 NPC 系统。未来的友方 NPC 不只是地图上的一个单位，而应该有自己的职业、性格、目标、资源、好感度和战斗能力。玩家可以和他们互动，也可以带着他们一起冒险。更进一步，重要 NPC 可以拥有独立记忆和简单决策能力，让世界显得更活。

### 角色养成与装备奖励

第三，会继续完善角色养成、背包、装备和奖励系统。跑团的乐趣不只是打赢一次战斗，还包括获得新装备、学习新法术、升级、选择专长和改变角色构筑。

### 更完整的战术地图

第四，会扩展战术地图。未来可以加入障碍物、地形、视线遮挡、范围模板、陷阱、机关和更清晰的移动交互，让战斗更接近桌面跑团。

### 线上多人同游

最后，长期目标是支持线上多人同游。多个玩家可以进入同一个 Session，各自扮演角色，由 AI DM 统一处理多人输入、行动顺序和场景调度。这样项目就不再只是一个单人 AI 游戏，而会更接近真正的线上跑团房间。

## 总结

TRPG-Agent 的核心价值在于把 AI 的自由叙事能力和程序的规则执行能力结合起来。AI 负责“讲故事”和“理解人话”，后端负责“守规则”和“记状态”，前端负责把过程变得可见、可玩、有反馈。

它现在还不是一个内容完全丰富的跑团平台，但已经搭出了很关键的骨架：Agent 流程、工具调用、状态持久化、战斗规则、法术效果、空间地图、角色面板和骰子动画。接下来只要继续补充冒险模组、NPC、装备成长和多人协作，它就有机会从技术原型变成一个真正能长期游玩的 AI 跑团系统。
